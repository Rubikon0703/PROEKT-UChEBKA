 # 🔌 Распиновка и схема подключения
## DS18B20 (1-Wire)
## Пин: GPIO 14
Назначение: Линия данных
Важные нюансы: Обязателен подтягивающий резистор 4.7 кОм между GPIO 14 и 3.3V. Паразитное питание не поддерживается.

## BH1750 (I2C)
## Пины: SDA: GPIO 8, SCL: GPIO 7
Назначение: Данные и тактирование
Важные нюансы: В коде включены внутренние pull-up. Если длина проводов превышает 15 см, добавьте внешние резисторы 4.7 кОм на 3.3V. I2C-адрес модуля: 0x23.

## HC-SR04 (Ультразвук)
## Пины: TRIG: GPIO 40, ECHO: GPIO 41
Назначение: Запуск импульса и приём эха
Важные нюансы: Модуль питается от 5V. Сигналы TRIG/ECHO также 5V. ESP32-S3 толерантен к 5В на большинстве пинов, но для долгосрочной надёжности рекомендуется использовать преобразователь уровней 3.3V↔5V.

## RC522 (RFID, SPI)
## Пины: SCLK: 12, MISO: 13, MOSI: 11, CS: 10, RST: 9
Назначение: Тактирование, ввод, вывод, выбор устройства, сброс
Важные нюансы: Питание 3.3V. Соединяется напрямую. Пин RST программно подтягивается к 1 при старте.

## Геркон (Датчик двери)
## Пин: GPIO 15
Назначение: Состояние двери (открыта/закрыта)
Важные нюансы: Схема подключения: один контакт геркона на GND, второй на GPIO 15. В коде включён GPIO_PULLUP_ENABLE. Логика инвертирована: замкнутый геркон (дверь закрыта) даёт 0, разомкнутый (дверь открыта) даёт 1.

# 🔍 Подробный разбор функций по файлам

## 🟢 app_main.c
void app_main(void)
Назначение: Точка входа ESP-IDF. Оркестрирует запуск подсистем и цикл опроса.
Как работает:
nvs_flash_init() → инициализирует энергонезависимую память для WiFi/MQTT параметров.
esp_netif_init() + esp_event_loop_create_default() → запускает сетевой стек и обработчик событий.
example_connect() → подключается к WiFi (SSID/пароль берутся из menuconfig).
sensors_init() → настраивает все шины и GPIO.
Запускает бесконечный цикл: вызывает sensors_read(), логирует данные в UART, ждёт 2 секунды через vTaskDelay().
Почему так: Разделение инициализации и рабочего цикла упрощает отладку и OTA-обновления. Задержка 2000 мс выбрана как компромисс: DS18B20 требует ~750мс на конвертацию, HC-SR04 не любит опрос чаще 50мс, а RTOS получает кванты времени на фоновые задачи (WiFi-стек, таймеры).
## 🔵 sensors.h / sensors.c

Публичный API

esp_err_t sensors_init(void)
Назначение: Настройка аппаратных интерфейсов и GPIO.
Как работает: Создаёт конфигурацию GPIO для TRIG/ECHO/OW/DOOR. Инициализирует I2C шину (i2c_new_master_bus) и добавляет устройство BH1750. Инициализирует SPI2, добавляет RC522. В режиме SIMULATION_MODE полностью пропускает инициализацию драйверов.
Почему так: Централизованная инициализация предотвращает конфликты при старте. Ошибки обрабатываются через ESP_ERROR_CHECK, чтобы система падала сразу при неправильной конфигурации пинов или занятой шине.

esp_err_t sensors_read(sensor_data_t *out)
Назначение: Атомарное чтение всех датчиков в одну структуру.
Как работает: Обнуляет структуру out, записывает текущий timestamp_ms. Последовательно вызывает функции чтения каждого датчика, включая read_door_state(). Заполняет поле rfid_uid временным значением, если карта обнаружена.
Почему так: Единый вызов упрощает логику в app_main.c. Если один датчик недоступен, остальные продолжают работать (возвращают -1.0f или false). Структура передаётся по указателю, избегая копирования в стеке задачи.

bool sensors_rfid_present(void)
Назначение: Быстрая проверка наличия карты.
Как работает: Вызывает rc522_check_card(), возвращает true/false.
Почему так: Оптимизация для сценариев, когда нужно мгновенно реагировать на карту без ожидания чтения температуры, света или состояния двери.

void sensors_deinit(void)
Назначение: Корректное освобождение ресурсов.
Как работает: Вызывает spi_bus_remove_device, i2c_master_bus_rm_device, i2c_del_master_bus, spi_bus_free.
Почему так: Предотвращает утечки памяти и конфликты при перезапуске задачи или OTA-обновлении. Порядок вызовов строго соответствует требованиям ESP-IDF v5.x.
Внутренние драйверы

float bh1750_read_lux()
Назначение: Чтение освещённости по I2C.
Как работает: Отправляет команду 0x10 (Continuous H-Resolution). Ждёт 180мс. Читает 2 байта. Применяет формулу: (MSB<<8 | LSB) / 1.2.
Почему так: BH1750 не имеет пина готовности данных. Polling с фиксированной задержкой гарантирует, что данные уже сконвертированы. Деление на 1.2 соответствует коэффициенту чувствительности из даташита.

float hc_sr04_read_cm()
Назначение: Измерение расстояния ультразвуком.
Как работает: Подаёт импульс 10 мкс на TRIG. Ждёт подъёма уровня на ECHO (таймаут 30мс). Замеряет длительность высокого уровня через esp_timer_get_time(). Считает расстояние: длительность_мкс * 0.0343 / 2.
Почему так: Используется высокоточный системный таймер, а не vTaskDelay, так как задержки RTOS не гарантируют микросекундную точность. Деление на 2 учитывает путь звука "туда-обратно".

void rc522_write_reg() / uint8_t rc522_read_reg()
Назначение: Обмен с RC522 по SPI.
Как работает: Формирует spi_transaction_t с флагами SPI_TRANS_USE_TXDATA/RXDATA. Адресный байт формируется так: бит 0 = 0 для записи или 1 для чтения. Отправляет в polling-режиме.
Почему так: Polling выбран вместо DMA/callback, так как RC522 опрашивается редко, а overhead RTOS на прерывания не оправдан для простого чтения регистра версии.

bool rc522_check_card()
Назначение: Проверка живости чипа и наличия карты.
Как работает: Читает регистр версии 0x37. Возвращает true, если значение равно 0x91, 0x92 или 0x12.
Почему так: Регистр версии не требует аутентификации. Значения соответствуют ревизиям чипа MFRC522. Полная проверка UID требует антиколлизии, которая здесь опущена для компактности кода.

ow_reset() / ow_write_bit() / ow_read_bit() / ow_write_byte() / ow_read_byte()
Назначение: Программная реализация протокола 1-Wire на GPIO.
Как работает: Используют esp_rom_delay_us() для жёстких таймингов: Reset (480мкс low), Write 0 (60мкс low), Write 1 (6мкс low → high), Read (2мкс low → sample через 8мкс).
Почему так: 1-Wire критичен к временным интервалам. ROM-задержки не прерываются планировщиком FreeRTOS, что гарантирует стабильность связи с DS18B20 даже при высокой загрузке CPU.

float ds18b20_read_temp()
Назначение: Чтение температуры.
Как работает: Выполняет сброс шины → отправляет 0xCC (Skip ROM) → 0x44 (Convert T) → ждёт 800мс → снова сброс → 0xCC → 0xBE (Read Scratchpad) → читает 9 байт → вычисляет (data[1]<<8 | data[0]) * 0.0625.
Почему так: Команда 0xCC игнорирует ROM-код, так как на шине предполагается один датчик. Конвертация занимает до 750мс при 12-bit разрешении. Множитель 0.0625 = 1/16 шагов LSB.

static bool read_door_state()
Назначение: Чтение состояния двери.
Как работает: Возвращает gpio_get_level(PIN_DOOR) == 1.
Почему так: Логика инвертирована относительно физического замыкания: внутренняя подтяжка даёт 1 при разрыве цепи (дверь открыта) и 0 при замыкании на GND (дверь закрыта). Это стандартная безопасная схема для герконов.


## 🟡 network.h / network.c

bool network_is_connected(void)
Назначение: Проверка наличия валидного IP-адреса.
Как работает: Запрашивает дескриптор интерфейса esp_netif_get_handle_from_ifkey("WIFI_STA_DEF") и "ETH_DEF". Если интерфейс существует и esp_netif_get_ip_info() возвращает ip.addr != 0, возвращает true.
Почему так: Не управляет подключением (это делает protocol_examples_common). Даёт бинарный ответ для условий вида if (network_is_connected()) publish(). Проверяет оба интерфейса для универсальности проекта.

## 🔴 mqtt_manager.h / mqtt_manager.c (Архив/Заглушка)

esp_err_t mqtt_init(uri, client_id)
Назначение: Создание и настройка клиента.
Как работает: Конфигурирует esp_mqtt_client_config_t, привязывает mosquitto.org.crt из EMBED_TXTFILES, регистрирует mqtt_event_handler, устанавливает Last Will Testament (LWT) offline.
Почему так: TLS обязателен для публичных брокеров. Встроенный сертификат исключает зависимость от файловой системы. LWT гарантирует очистку статуса на брокере при обрыве связи.

esp_err_t mqtt_connect(void)
Назначение: Установка соединения.
Как работает: Вызывает esp_mqtt_client_start(). Блокирует задачу в цикле while(!connected) с таймаутом ~5с. При успехе публикует online.
Почему так: Синхронное ожидание упрощает логику старта. В продакшене лучше использовать асинхронный подход или выносить в отдельную задачу.

esp_err_t mqtt_publish_sensor_data(data, topic)
Назначение: Отправка JSON-пакета.
Как работает: Создаёт cJSON объект, заполняет поля, сериализует в строку, вызывает esp_mqtt_client_publish() с QoS 1. Освобождает память.
Почему так: JSON стандартен для IoT. QoS 1 гарантирует доставку хотя бы один раз. cJSON_PrintUnformatted экономит трафик и память.

static esp_err_t mqtt_event_handler(event)
Назначение: Callback событий MQTT.
Как работает: Обрабатывает CONNECTED, DISCONNECTED, PUBLISHED, ERROR. Управляет флагом mqtt_connected.
Почему так: Событийная модель ESP-MQTT требует callback. Разделение состояний позволяет безопасно публиковать данные только при активном соединении.

mqtt_disconnect() / mqtt_deinit()
Назначение: Корректное завершение работы.
Как работает: Публикует offline, останавливает клиент, разрушает handle.
Почему так: Гарантирует чистое отключение от брокера и освобождение кучи.
